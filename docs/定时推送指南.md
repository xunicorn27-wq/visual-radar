# 定时推送指南

## 当前运行方式

Visual Radar 当前采用无 OpenAI API Key 的半自动流程：由 Codex 协助完成采集结果分析和日报生成，人工检查后再提交数据。GitHub Pages 只发布仓库中已经生成的静态内容，不运行分析任务，也不负责企业微信发送。

`.github/workflows/daily.yml` 仅保留 `workflow_dispatch` 人工入口。每次运行都固定请求：

```text
POST $DEPLOYED_URL/api/automation/daily?dryRun=1
```

该入口只用于人工验证已部署服务的日报流程，始终是 dry-run，不存在关闭 dry-run 或正式发送的分支。当前没有 `schedule` 或 `cron`，不会无人值守执行。

## 人工 Dry Run

需要验证部署服务时：

1. 确认公网 Express 服务可用，且 `/api/health` 返回成功。
2. 在 GitHub Actions Secrets 配置 `DEPLOYED_URL`。
3. 配置与部署环境一致的 `CRON_SECRET`。
4. 打开 `Actions` → `Visual Radar Manual Dry Run` → `Run workflow`。
5. 检查工作流和部署日志中的 dry-run 结果。

这个工作流不接受输入参数，也不能通过 Actions 页面切换为正式发送。企业微信正式发送必须继续由受控的人工流程执行，并在发送前检查 Pages 页面和 dry-run 内容。

## GitHub Pages 的边界

Pages 工作流只负责构建和发布静态站点：

- 不调用 OpenAI API。
- 不生成新的分析或日报。
- 不触发 daily API。
- 不发送企业微信消息。

静态内容必须先通过 Codex 半自动流程生成、人工检查并提交，之后才由 Pages 发布。

## 恢复自动计划的条件

不要直接把 `schedule` 或 `cron` 加回 `daily.yml`。恢复无人值守 daily 前必须重新评审，至少确认：

- 已有可用的 OpenAI API Key 和稳定的公网 Express 部署。
- 分析、日报生成、失败处理和企业微信发送链路适合无人值守运行。
- 密钥权限、调用成本、重复发送和异常重试风险已经评估。
- 工作流 contract test 和操作文档同步更新并通过审查。

完成重新评审前，daily 必须保持仅人工 dry-run。
