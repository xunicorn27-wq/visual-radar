# 定时推送指南

## 当前状态：旧 Daily Workflow 已停用

`.github/workflows/daily.yml` 已删除。当前无 OpenAI API Key、无公网 Express 服务，GitHub Actions 不调用 `/api/automation/daily`，也不需要配置 `DEPLOYED_URL` 或 `CRON_SECRET`。

仓库中的 GitHub Actions 仅负责静态 Pages：测试、类型检查、构建仓库中已经生成的静态数据，并发布站点。它不采集、不分析、不生成日报，也不发送企业微信消息。

## 为什么 Daily Dry Run 不能用于预览

`/api/automation/daily?dryRun=1` 中的 `dryRun` 只跳过最后的企业微信 publish。调用仍会依次执行：

1. collect：采集并写入内容。
2. analyze：调用分析 Provider。
3. generate：生成并写入日报。
4. 跳过 publish，不发送企业微信。

因此，无 API Key 时该接口会在分析阶段失败，不能作为无副作用的日报或企业微信预览入口。当前阶段不应从 GitHub Actions 或其他无人值守任务调用它。

## 当前 Codex 半自动流程

当前日报流程在本地受控执行：

1. 本地采集候选内容并生成 Agent batch。
2. 使用 Codex 将结果写入 `data/visual_radar_agent_output.json`，不依赖 OpenAI API Key。
3. 运行 `pnpm agent:import`，再在本地生成日报并检查 issue 内容。
4. 先运行测试和 `build:pages`，再将确认后的 data 分析和 issues 推送到 `main`。
5. 等待 GitHub Pages 工作流构建并发布静态站点。

Pages 发布成功后，应先打开对应 issue 页面确认内容和链接可访问，再进入企业微信预览或人工发送步骤。

## 企业微信 Markdown 预览

预览前必须已经存在日报 issue，并且 Pages Action 已成功、实际 deployment URL 已替换到本地 `.env.local`、手机已验证详情页。运行：

```bash
pnpm wecom:preview -- 2026-07-16
```

将日期替换为已有本地日报 ID。脚本读取 `.env.local` 中的 `VISUAL_RADAR_PUBLIC_URL` 和本地 issue 文件，直接生成企业微信 Markdown 并返回 `sent: false`。它不启动 Express、不读取 `CRON_SECRET` 或 Webhook、不访问网络，也不会重新采集、分析、生成或发送日报。

只有单期 preview Markdown 已检查并且用户明确确认后，才能从本地管理区或明确的不带 dry-run 的正式接口发送。未替换占位 URL、未以 Actions 输出为准或未验证公网详情时不得发送。

## 恢复自动计划的条件

不要直接恢复旧 `daily.yml` 或新增 `schedule`。恢复无人值守 daily 前必须同时具备：

- 可用且权限、额度明确的 OpenAI API Key。
- 稳定、可观测的公网 Express 部署。
- 部署端和 GitHub Actions 所需 secrets 的完整配置与轮换方案。
- 分析、日报生成、失败处理和企业微信发送链路适合无人值守运行。
- 密钥权限、调用成本、重复发送和异常重试风险已经评估。
- 新工作流、contract test 和操作文档经过重新安全评审。

完成这些条件和评审前，GitHub 必须保持仅做静态 Pages 发布。
